### 4. Лимит 30 вакансий за сеанс (ПРАВИЛЬНЫЙ подход)
Нужно изменить архитектуру: `dashboard-server.mjs` должен отслеживать глобальный собранный лимит и передавать каждому процессу `scan-telegram.mjs` оставшийся лимит.

**Файл:** `dashboard/public/app.js` (строка ~468)
```javascript
// Было:
const scanLimit = 10;

// Стало: передаем 30 как глобальный лимит
const scanLimit = 30;
```

**Файл:** `scripts/dashboard-server.mjs` — добавить трекинг глобального лимита:
```javascript
// В sourcingProgress добавить:
sourcingProgress.globalLimit = scanLimit; // 30
sourcingProgress.collectedCount = 0; // счетчик собранных

// При запуске каждого scan-telegram.mjs:
const remainingLimit = Math.max(0, scanLimit - sourcingProgress.collectedCount);
if (remainingLimit <= 0) {
  // Прекратить поиск
  break;
}
env.HH_SCAN_LIMIT = String(remainingLimit); // передать оставшийся лимит
```

**Файл:** `scripts/scan-telegram.mjs` — оставить как есть, он принимает `HH_SCAN_LIMIT` из env.

**Примечание:** Deduplication вакансий происходит в `addVacancyRecord()` — если URL уже существует, он не добавляется повторно. Это уже ограничивает общее количество уникальных вакансий.

**Логика завершения при глобальном лимите:**
1. При достижении `collectedCount >= 30` — отменить pending процессы и завершить
2. Не нужно пытаться "честно" распределять 30 между keywords — просто стоп когда набрали 30
3. Progress bar покажет реальное количество собранных (может быть чуть больше 30 если последний процесс уже запущен с >0 вакансий)
